# 00 Sの思考・方向性・判断基準

Claude CodeおよびAIとの作業において、
Sが毎回説明しなくて済むよう、思考・方向性・判断基準を定義する。

このファイルを読めばSが「浅い」「違う」「もっと深く」と
言いたくなることを事前に防げる。

---

## Sという人間

```
30代中盤・クリエイティブプロダクション勤務・CD
アナログ→デジタル→AIの変化を現場で全部経験してきた
5ヶ月の娘がいる・愛知県在住
座右の銘：「下の者ができない原因は、上の者にある」

今やってること：
AIで仕事に余白を作る。
その余白を自分時間・家族・本当にやりたいことに使う。
これを発信しながらnoteでマネタイズする。
```

---

## Sが「浅い」と感じるパターン

```
① 俺が言ったことをそのまま整理しただけの提案
  → ブラッシュアップがない
  → 「なんで自分で言ったことを返されてるんだ」となる
  → 必ず「プロとしての視点」を上乗せすること

② 表面の課題だけ解決する提案
  → 「具体的にどうすればいいかわからない」の前に
     「変われるかどうかわからない」という深層がある
  → 表層・中層・深層の3層で考えること

③ 説明しすぎるプロフィール・キャッチコピー
  → 言わないけど滲み出る、が正解
  → 「AIで余白を作る」と言わずに「余白」が伝わる設計

④ 「〜だと思います」で終わる提案
  → 断言して、カウンターを入れる
  → Sの文体と同じ構造で提案する

⑤ フェーズを無視した提案
  → 今はフェーズ1（立ち上げ期）
  → noteへの誘導・マネタイズの話は条件が揃ってから
  → data/phase.jsonのnote_enabledを必ず確認すること
```

---

## 提案する時の基準

```
Sに何かを提案する時は以下を必ず通す：

① 「ターゲットはS自身」かどうか
  → Sが見て「これ俺のことだ」と思えるか

② 「具体的に何をすればいいか」があるか
  → プロンプト・手順・ツール・数字・結果のいずれか必須
  → 「なんとなくいい話」で終わってないか

③ 「俺感」があるか
  → 説明臭くないか
  → キラキラしてないか
  → カウンター構造があるか（強い言葉の後に余白を作る）

④ 現場で使えるか
  → 制作現場のリアルに即してるか
  → 「代理店ソルジャー」「カンプ」「オリエン」が
     自然に出てくるレベルのリアリティがあるか
```

## 設計に対するSの姿勢

```
「完璧を目指してから動く」ではなく
「動かしながら完璧に近づける」

ただし：
「運用前に直せることは直しておく」
→ 設計倒れで動かすのは嫌
→ 動く状態で精度を上げていく

優先順位：
① 動くこと
② 精度が上がること
③ コストが下がること
```

## 横展開の考え方

```
大枠の仕組みは変えない：
├── エージェント6体の構成
├── 承認UIの設計
├── フィードバックループ
└── フェーズ管理

切り口ごとに変えるもの：
├── キャラクター・一人称・文体
├── ターゲットの定義（3層）
├── 投稿の型の具体例
├── 登場する関係者（シーンマップ）
├── NGワード・表現
└── リサーチャーの収集テーマ

「コードは共通・ナレッジを差し替えるだけ」が原則
```

## Sがやりたくないこと

```
① 毎回同じ説明を繰り返すこと
  → このファイルを読んでいれば不要なはず

② 浅い提案にフィードバックを返し続けること
  → 最初からSの水準で提案してほしい

③ 設計と実装がズレること
  → doc/10_active_rules.mdが最優先
  → 設計書を読まずに実装しないこと

④ フェーズを無視したコンテンツを出すこと
  → 「noteに置いてます」はnote_enabled: trueになってから

⑤ 横展開の時に一から説明し直すこと
  → このファイルとdoc/11_expansion_guide.mdを読めば不要
```

## AIへの指示スタイル

```
Sは指示が短い・簡潔
→ 「これをもっとよくして」「なんか違う」「浅い」

これらの言葉が出た時の解釈：

「浅い」→ 表層だけ触ってる・深層まで考えられてない
「なんか違う」→ Sの文体・トーン・現場感が出てない
「もっとよくして」→ Sが言ったことをそのまま返してる
「俺感がない」→ 説明臭い・キラキラしすぎ・断言しすぎ
「フェーズじゃない」→ data/phase.jsonを確認すること

これらのフィードバックが来る前に
Sの水準で提案できるようにすること
```
