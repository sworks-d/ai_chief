# 07 モデル割り振り・コスト最適化設計

## 基本方針

```
「頭の良さが必要な作業にだけ高いモデルを使う」

Haiku：定型作業・データ取得・APIを叩くだけの処理
Sonnet：文章生成・分析・判断が必要な処理
Opus：使わない（コスパが悪すぎる）
```

## 各エージェントのモデル割り振り

### エージェント1：リサーチャー → **Haiku**

```javascript
model: "claude-haiku-4-5-20251001"
```

**理由：**
```
やること：
├── URLを取得してテキストを読む
├── 決まったフォーマットで要約する
├── テーマタグ・ソース種別を付与する
└── JSONで出力する

判断の難しさ：低
→ ルールが明確・フォーマットが決まってる
→ Haikuで十分な品質が出る
→ 1日30件処理してもSonnetの1/3のコスト
```

**推定コスト（1日）：**
```
処理量：約5万トークン
Haiku：$0.04
Sonnet：$0.20
削減効果：$0.16/日（月$4.8削減）
```

---

### エージェント2：ファクトチェッカー → **Haiku**

```javascript
model: "claude-haiku-4-5-20251001"
```

**理由：**
```
やること：
├── 鮮度スコアの判定（A/B/C/D）
├── 信頼度スコアの判定（HIGH/MID/LOW）
├── 有益性の判定
└── TYPE分類（A/B/C/D）の決定

判断の難しさ：低〜中
→ 判定ルールが明確に定義されている（doc/04_agents.md参照）
→ 「このソースはHIGHか？」はルールベースで判断できる
→ Haikuで十分

ただし注意：
→ 「両論提示のヒント生成」部分はSonnetの方が精度が上がる
→ コスト優先ならHaiku・品質優先ならSonnet
→ まずHaikuで試して品質が足りなければSonnetに変更
```

**推定コスト（1日）：**
```
処理量：約3万トークン
Haiku：$0.02
Sonnet：$0.12
削減効果：$0.10/日（月$3削減）
```

---

### エージェント3：ライター → **Sonnet（絶対に下げない）**

```javascript
model: "claude-sonnet-4-6"
```

**理由：**
```
やること：
├── Sの文体・口調を完全に再現する
├── カウンター構造を自然に入れる
├── 制作現場の言葉を適切に使う
├── 3層構造（ソース→解釈→有益性）を組む
├── 品質スコア10項目で自己採点する
└── シーンが浮かぶ締め方を作る

判断の難しさ：高
→ 「Sっぽさ」の再現はモデルの品質に直結する
→ Haikuにすると「AIが書いた感」が出る
→ ここだけは絶対にSonnetを維持する

収益への直結度：最高
→ ライターの品質 = フォロワーが増えるかどうか
→ フォロワーが増えるかどうか = noteが売れるかどうか
→ ここにケチると全体の収益が落ちる
```

**推定コスト（1日）：**
```
処理量：約4万トークン（X3本+Threads2本生成）
Sonnet：$0.16
→ ここは削減しない
```

---

### エージェント4：ポスター → **Haiku**

```javascript
model: "claude-haiku-4-5-20251001"
```

**理由：**
```

├── キューから投稿文を読み込む

└── 結果をログに記録する

判断の難しさ：ほぼゼロ
→ コードを実行するだけ
→ 文章を考える必要がない
→ Haikuで十分どころかHaikuが最適
```

**推定コスト（1日）：**
```
処理量：約0.5万トークン
Haiku：$0.004
Sonnet：$0.02
削減効果：小さいが積み重なる
```

### エージェント5：フェッチャー → **Haiku**

```javascript
model: "claude-haiku-4-5-20251001"
```

**理由：**
```
やること：
├── X APIを叩いてメトリクスを取得
├── Threads APIを叩いてメトリクスを取得
├── JSONに整形して保存
└── 1h・6h・24hのタイミングで3回実行

判断の難しさ：ほぼゼロ

→ Haikuが最適
```

**推定コスト（1日）：**
```
処理量：約0.5万トークン × 3回 = 1.5万トークン

→ 最小コスト
```

---

### エージェント6：アナリスト → **Sonnet**

```javascript
model: "claude-sonnet-4-6"
```

**理由：**
```
やること：
├── 1週間分のメトリクスデータを分析する
├── 「なぜ伸びたか」の仮説を立てる
├── ターゲット人物像を浮かび上がらせる
├── 次の投稿への改善指示を生成する
└── ライターとリサーチャーへのフィードバックを出す

判断の難しさ：高
→ パターン認識・仮説立案・戦略的判断が必要
→ ここの精度がシステム全体の改善速度を決める
→ Haikuにすると分析が浅くなりフィードバックの質が落ちる

ただし：
→ 日次分析（毎日）：Haiku可（単純な数字の比較）
→ 週次・月次分析：Sonnet必須（深い洞察が必要）
→ 将来的に分けて設計するとさらにコスト削減できる
```

**推定コスト（1日）：**
```
処理量：約3万トークン
Sonnet：$0.12
→ 週次・月次分析は週1・月1なのでコスト影響は小さい
```

## 実装方法

### モデルの指定方法（コード例）

```javascript
// agents/researcher.js
const response = await anthropic.messages.create({
  model: "claude-haiku-4-5-20251001",  // Haiku
  max_tokens: 2000,
  messages: [...]
});

// agents/writer.js
const response = await anthropic.messages.create({
  model: "claude-sonnet-4-6",  // Sonnet（変えない）
  max_tokens: 4000,
  messages: [...]
});
```

### プロンプトキャッシュの有効化（コード例）

```javascript
// ナレッジファイルをキャッシュ対象にする
const response = await anthropic.messages.create({
  model: "claude-sonnet-4-6",
  max_tokens: 4000,
  system: [
    {
      type: "text",
      text: knowledgeContent,  // ナレッジファイルの内容
      cache_control: { type: "ephemeral" }  // キャッシュ指定
    }
  ],
  messages: [...]
});
```

## モニタリング方法

実際にいくらかかってるかを確認する方法：

```bash
# Anthropic Consoleで確認
https://console.anthropic.com/settings/cost

# ローカルでトークン数を確認（実装後）
node scripts/check_usage.js
```

月の予算上限を設定しておくと安心：
```
console.anthropic.com/settings/limits
→「Monthly spend limit」を$20に設定
→ 超えたら自動でAPIが止まる
```
